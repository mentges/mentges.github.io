I Integrated AI Agents Into Our Data Pipelines. Here's What Actually Worked.

Lessons from bringing agentic AI into production data engineering at a major B2B platform — what shipped, what didn't, and what I'd do differently.

Author Bruno Mentges
Published April 2026
Reading time 12 min

There's a gap between "we use AI" and "AI is part of how we engineer." Most data teams I talk to are somewhere in the first camp — individual engineers using chat interfaces for code generation, maybe a copilot plugin, occasionally pasting error logs into a prompt. That's useful, but it's not integration. It's augmentation at the individual level, with no team-wide impact on how work gets done.

Over the past two years, I pushed further at a major B2B data platform. I integrated LLM-powered agents directly into our data engineering workflows — not as a side tool, but as a core part of how we built, maintained, and operated pipelines processing terabytes daily. Some of it worked remarkably well. Some of it failed in instructive ways.

This is the honest account.

The starting point: what we had

The data engineering team maintained hundreds of pipelines across multiple cloud providers, orchestrated by a standard workflow engine, with a transformation framework handling the modeling layer, and a cloud analytical warehouse as the serving layer. Terabytes of Sales and Finance data flowed through daily. Monitoring was handled by a data observability platform and incident management tooling.

The team was strong — senior engineers who knew the codebase well. But like every data team, we had the same recurring pain points:

  • Boilerplate fatigue — new pipelines required scaffolding that followed the same patterns every time, but with enough variation that you couldn't just copy-paste
  • Context switching on incidents — when a pipeline failed at 3 AM, the on-call engineer had to reconstruct context about a pipeline they might not have touched in months
  • Documentation drift — pipeline docs were always slightly behind the code, because updating docs is the thing that gets cut when you're shipping under pressure
  • Code review bottlenecks — senior engineers spent significant time reviewing PRs for the same categories of issues: missing null handling, partition strategy mistakes, schema drift risks

These aren't exotic problems. Every data team has them. The question was whether AI agents could address them at the workflow level, not just the individual prompt level.

What I built

I didn't start with a grand plan. I started with the pain points above and worked backward to what an agent would need to be useful for each one. The key insight came early: a generic LLM is almost useless for data engineering work. A context-rich agent is transformative.

1. Model Context Protocol changed everything

The single biggest lever was MCP — Model Context Protocol. Before MCP, using an AI for our specific codebase meant either dumping thousands of lines into a prompt (expensive, noisy, often exceeding context limits) or accepting generic advice that didn't account for our patterns.

MCP let me set up structured context servers that gave agents access to exactly what they needed:

  • Pipeline metadata — DAG definitions, schedule configurations, dependency graphs, recent run histories
  • Schema information — warehouse table schemas, column types, freshness metadata, lineage
  • Transformation model definitions — model SQL, documentation, test definitions, upstream/downstream dependencies
  • Team conventions — naming patterns, partition strategies, testing requirements, PR checklist

With this context, agents stopped giving generic Python advice and started giving answers grounded in our infrastructure. "The upstream model refreshes at 06:00 UTC, so your dependency should wait until 06:30 to account for the 15-minute SLA buffer we use" — that kind of specificity.

2. Pipeline scaffolding agents

The first agent I shipped was the simplest: a pipeline scaffolder. You describe what data you need to move and where, and the agent generates a pipeline following our exact patterns — correct naming conventions, proper partition strategy, standard monitoring hooks, tests pre-populated.

This sounds trivial, but the compound time savings were significant. A new pipeline that took 45-60 minutes to scaffold correctly (including looking up conventions you half-remember, copying from a similar pipeline, adjusting the parts that are different) now took 5 minutes of describing what you want plus 10 minutes of review and adjustment.

What worked: Boilerplate generation, naming convention enforcement, test scaffolding. The agent was faster and more consistent than copying from an existing pipeline.

What didn't: Complex orchestration logic. When pipelines had non-standard dependency patterns or conditional execution paths, the agent would generate something plausible-looking but subtly wrong. We learned to treat scaffolded pipelines as a starting point, not a finished product — which is exactly the right mental model.

3. Code review assistants

The second agent ran as a pre-review step before human code review. It checked PRs for the categories of issues our senior engineers spent the most time catching:

  • Missing null/empty handling on columns that historically had nulls
  • Partition strategies that would cause skew at our data volumes
  • Schema changes that could break downstream consumers
  • Missing or insufficient test coverage for new logic
  • Deviation from team naming conventions

What worked: Catching mechanical issues. The agent was excellent at "you added a new column but didn't add a null check, and this source historically has 3.2% null rate" — factual, specific, actionable. Senior engineers reported spending less time on mechanical review and more time on architectural feedback.

What didn't: Architectural judgment. The agent couldn't evaluate whether a pipeline's overall design was right for the problem. It could catch that you forgot a null check but not that you should be using an incremental model instead of a full refresh. Human review remained essential for design-level feedback.

4. Incident response context

This was the most impactful agent for on-call quality of life. When a pipeline failed, the agent could immediately pull together:

  • The pipeline's recent run history (was this a first-time failure or recurring?)
  • Upstream pipeline status (did a dependency fail first?)
  • Recent code changes to the pipeline (did someone just modify it?)
  • Historical failure patterns (has this pipeline failed with similar errors before, and how was it resolved?)

Instead of the on-call engineer spending 15-20 minutes reconstructing context at 3 AM, the agent assembled a briefing in seconds. The engineer's job shifted from "figure out what's going on" to "confirm the diagnosis and act."

What worked: Context assembly. The agent's ability to pull together information from multiple systems (orchestrator, warehouse, version control, observability platform) into a coherent briefing was genuinely transformative for incident response time.

What didn't: Root cause analysis for novel failures. For patterns it had seen before, the agent was excellent. For genuinely new failure modes, it would sometimes suggest fixes that addressed symptoms rather than root causes. We added a confidence score to agent suggestions and trained the team to be skeptical of low-confidence recommendations.

5. Documentation generation

The documentation agent was a partial success. It could generate accurate descriptions of what a pipeline does by reading the code, and it kept docs in sync when code changed. But the valuable documentation — why a pipeline exists, what business decisions it supports, what the gotchas are — still required human input.

We settled on a hybrid: the agent generates and maintains the "what" (schema descriptions, data flow, schedule, dependencies), and humans write the "why" (business context, known issues, operational notes). This eliminated the most tedious part of documentation while preserving the most valuable part.

The three principles that made it work

Looking back, the difference between the agents that shipped and stayed versus the ones we abandoned came down to three principles:

1. Guardrails first, capabilities second

Before expanding what an agent could do, I defined what it couldn't do. Agents could read production data but not modify it. They could suggest pipeline changes but not deploy them. They could generate code but not merge it. Every capability had an explicit boundary.

This sounds limiting, but it's what made adoption possible. Engineers trusted the agents because the blast radius of a bad suggestion was zero — the worst case was wasted time reading a bad recommendation, not a production incident caused by an autonomous agent.

2. Context beats intelligence

A less capable model with rich, specific context consistently outperformed a more capable model with generic context. The MCP servers I built were more important than the choice of LLM. When the agent knew our specific schemas, conventions, and history, even simple reasoning produced useful output. Without that context, even sophisticated reasoning produced generic advice.

If you're investing in AI for your data team and you're debating which model to use, stop. Invest in the context layer instead. That's where the value is.

3. Workflow integration, not feature addition

The agents that stuck were the ones that fit into how engineers already worked. The pipeline scaffolder produced code in the same format engineers were already writing. The code review assistant showed up in the same PR interface engineers were already reviewing. The incident responder appeared in the same alerting channel engineers were already monitoring.

The agents that failed were the ones that required a new workflow. An early experiment with a "data exploration agent" that engineers could query in a separate interface got almost zero adoption — not because it was bad, but because it required engineers to leave their normal flow.

What I'd do differently

If I were starting this from scratch today:

  1. Start with incident response, not code generation. The incident context agent had the highest impact-to-effort ratio and the fastest team buy-in. Code generation is where everyone starts, but incident response is where the pain is worst.
  2. Build the context layer before the agent layer. I spent the first month building MCP servers and context infrastructure before writing a single agent. That felt slow at the time, but every agent built on top of that context was immediately useful. Teams that skip this step end up with agents that give generic advice.
  3. Measure adoption, not capability. The most impressive agent demo doesn't matter if no one uses it. I should have instrumented adoption metrics earlier. The agents that looked best in demos (data exploration) had the lowest actual usage. The boring ones (scaffolding, documentation) had the highest.
  4. Set explicit expectations about what agents can't do. The biggest source of disillusionment was when engineers expected agents to handle architectural decisions. Setting the frame early — "this is a power tool, not an architect" — would have prevented some early frustration.

The bottom line

AI agents in data engineering are real and production-ready — but only if you treat them as structured tools within a workflow, not as magic boxes. The value comes from three things: rich context (MCP), clear guardrails (what the agent can't do), and seamless integration (agents meet engineers where they already work).

The teams that will get the most from AI aren't the ones with the most advanced models. They're the ones that invest in the context infrastructure and workflow integration that makes AI useful for their specific environment.

AI doesn't replace data engineers. It amplifies their judgment. And the difference between "we use AI" and "AI is part of how we engineer" is whether you've built the infrastructure to make that amplification systematic.
